# Architecture Document

## Project

**Grandeur Properties Global Listing Intelligence Pipeline**

## Architecture Overview

This Microsoft Fabric pipeline is designed to ingest property listing files produced by multiple regional offices and load them into a centralized Lakehouse table for analytics and reporting. The solution uses a file-based ingestion pattern, where source CSV files are dropped into a landing area and processed automatically by a Fabric pipeline.

The architecture emphasizes four core objectives:

- Standardized ingestion from multiple offices
- Incremental loading of new and changed property records
- Privacy-aware curation through exclusion of PII fields
- Reliable operational handling of processed source files

At a high level, the design consists of a landing zone for incoming files, a pipeline with three sequential activities, and a Lakehouse target table that stores the curated property listing data.

## Source and Landing Layer

Regional offices export property listing data as CSV files and drop them into the `landing/files` area. These files follow a naming convention such as:

- `office_london.csv`
- `office_newyork.csv`
- `office_dubai.csv`

The pipeline is configured to read files using the wildcard pattern `office_*.csv`. This allows Microsoft Fabric to pick up all office files that follow the agreed naming standard without requiring separate pipeline branches or office-specific logic.

This landing layer serves as the raw intake point for the pipeline. It isolates inbound source files from curated analytical storage and provides a controlled place to manage file movement, archival, and deletion.

## Ingestion Pattern

The first pipeline activity is a **Copy activity**. Its responsibility is to read all matching files from the landing area and load the curated dataset into a Lakehouse table.

### Copy Activity Responsibilities

- Read source files from `landing/files`
- Match files using the wildcard pattern `office_*.csv`
- Load approved columns into the Lakehouse target table
- Exclude PII columns from the target dataset
- Add an ingestion timestamp generated by the pipeline
- Apply Upsert logic using `property_id` as the business key

The wildcard-based read pattern ensures the solution can process multiple office drops in a single design. Instead of maintaining individual ingestion logic for each office, the pipeline applies one standard ingestion rule set across all matching files.

## Lakehouse Target Design

The destination is a Microsoft Fabric Lakehouse table that stores the unified property listing dataset. This table acts as the curated analytical layer for downstream use cases such as executive reporting, property intelligence dashboards, and operational analysis.

### Upsert Strategy

The load uses **Upsert** behavior with `property_id` as the key. This means:

- If a `property_id` does not already exist in the target table, the record is inserted
- If a `property_id` already exists, the incoming record updates the existing one

This is important because office files may contain both newly listed properties and updates to previously loaded properties. Using Upsert prevents unnecessary full reloads and helps maintain a current, consolidated view of listings.

## Metadata and Privacy Controls

### Ingestion Timestamp

The pipeline adds an ingestion timestamp during processing. This timestamp records when the pipeline loaded the record into the Lakehouse, rather than relying only on the source file creation date.

This metadata supports:

- Operational traceability
- Auditability of data refresh cycles
- Easier troubleshooting for late or missing file deliveries
- Better downstream understanding of data recency

### PII Exclusion

Personally identifiable information (PII) columns are excluded during the ingestion step and are not loaded into the Lakehouse target table. This keeps the analytical dataset aligned with privacy and governance expectations.

Excluding PII at pipeline load time is preferable to loading it first and removing it later, because it reduces exposure risk and keeps the curated layer intentionally scoped to analytics-ready business attributes.

## File Lifecycle Architecture

After the data load completes, the pipeline performs two operational file-handling activities.

### Activity 2: Archive Processed Files

The second activity archives the processed source files from the landing area into an archive location. This creates a historical record of what was received and processed.

Archiving supports:

- Traceability of source deliveries
- Recovery and investigation if downstream issues are discovered
- Controlled reprocessing when necessary
- Better operational governance over inbound file history

### Activity 3: Delete Landing Files After Archive Success

The third activity deletes files from the landing area, but only after the archive step succeeds.

This dependency is intentional. Deleting landing files before confirming a successful archive would introduce avoidable data loss risk. If the archive step failed after deletion, the original source file would no longer be available for recovery, validation, or replay.

## Why Archive-Before-Delete Matters

The archive-before-delete pattern is a key reliability control in file-based pipelines.

Its importance is operational as well as architectural:

- It preserves a recoverable copy of the original source file
- It reduces the risk of permanent data loss during partial pipeline failures
- It supports replay scenarios if transformation or loading issues are later identified
- It provides evidence of what was received from each office and when
- It keeps the landing zone clean without sacrificing traceability

In short, the landing area should be treated as a transient intake zone, but files should not be removed from it until a safe retained copy exists in the archive layer.

## Why Wildcard Ingestion Improves Scalability

The wildcard design using `office_*.csv` makes the pipeline easier to scale as the business adds new offices.

For example, if Grandeur Properties opens a new Singapore office and begins dropping a file named `office_singapore.csv`, the existing pipeline can ingest it immediately as long as it follows the established naming convention and schema. No pipeline redesign, no new branch, and no office-specific activity is required.

This approach improves onboarding because it:

- Reduces maintenance overhead
- Avoids repeated configuration changes for each new office
- Enforces a standard contract for source delivery
- Makes the solution more extensible for global expansion

This is especially valuable in enterprise data platforms where the number of contributing business units can grow over time.

## End-to-End Pipeline Sequence

The architecture can be summarized as the following sequential flow:

1. Office CSV files are dropped into `landing/files`
2. The pipeline reads all files matching `office_*.csv`
3. Copy activity loads curated data into the Lakehouse table
4. Upsert logic uses `property_id` to insert or update records
5. The pipeline adds an ingestion timestamp to each record
6. PII columns are excluded from the analytical target
7. A second activity archives the processed files
8. A third activity deletes files from landing only after archive success

## Design Benefits

This architecture provides several practical benefits for a Microsoft Fabric implementation:

- Centralized ingestion across multiple offices
- Incremental updates without full-table replacement
- Improved governance through PII exclusion
- Stronger operational auditability with ingestion timestamps
- Safer file handling through archive-first cleanup
- Better extensibility through wildcard-based source onboarding

## Conclusion

The Grandeur Properties Global Listing Intelligence Pipeline demonstrates a practical Microsoft Fabric architecture for global file-based ingestion. It combines standardized CSV intake, curated Lakehouse loading, Upsert-based record management, privacy-aware design, and safe file lifecycle controls.

The result is a scalable and portfolio-ready architecture that reflects real-world data engineering priorities: maintainability, auditability, privacy, and operational resilience.
